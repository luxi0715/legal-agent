# M5 · 两阶段检索(Reranker)评测报告

> 从 M4 Hybrid Search 到 M5 Two-Stage Retrieval,一次完整的工程评测.
>
> 数据规模:25,211 条法律条文 chunks(344 部国家级法律)
> 评测代码:`scripts/compare_rerankers.py`

---

## 1. 实验设置

| 维度 | 配置 |
|---|---|
| 数据规模 | 25,211 条 chunks |
| 召回 — Vector | pgvector + HNSW(m=16, ef_construction=64) |
| 召回 — BM25 | PostgreSQL tsvector + GIN,jieba 中文分词 |
| 召回融合 | Reciprocal Rank Fusion (k=60) |
| 精排模型 | DashScope `gte-rerank-v2`(Cross-encoder API) |
| 精排策略 | Hybrid Top 50 → Reranker → Top 5 |
| 上下文重排 | Lost in the Middle 修复(奇偶位交错) |
| 拒答策略 | 三信号组合(绝对阈值 + 召回辅助 + 领域单一) |

## 3. 关键发现

### 发现 1:Reranker 显著改善"概念性查询"的排序质量

Query "什么是合同违约责任?":

| 方法 | Top 1-3 命中 |
|---|---|
| Vector (M3) | 农村土地承包法 / 民法典577 / 民法典582 |
| Hybrid (M4) | 民法典566 / 民法典495 / 建筑法15 |
| **Two-Stage (M5)** | **民法典522 / 建筑法15 / 民法典592** |

→ Reranker 把民法典违约相关核心条款(522/582/592)拉到 Top 5,
  Hybrid 阶段这些条款排在 50+ 位.

### 发现 2:Reranker 在"语义模糊"查询时具备语义补全能力

Query "老板拖欠工资怎么办?":

| 方法 | Top 5 法律覆盖 |
|---|---|
| Hybrid (M4) | 仅民事维权(劳动合同法 + 仲裁法 + 工会法) |
| **Two-Stage (M5)** | **多了刑法第276条(恶意欠薪罪)** |

→ Reranker 自动把 "拖欠工资"扩展到刑事责任,
  这是 Cross-encoder 的语义理解优势.

### 发现 3:多领域查询的"刑民交叉"识别

Query "合同诈骗的法律责任":

Two-Stage Top 5 同时命中:
- 刑法 第406条 / 224条 / 167条(刑事)
- 民法典 第500条 / 592条(民事)

→ 这是律师视角的标准答案.

### 发现 4:精确条款号查询 — 三种方法全军覆没 ⚠️

Query "民法典第五百七十七条规定了什么?":

| 方法 | Top 1 | 命中"民法典 第577条" |
|---|---|---|
| Vector (M3) | 循环经济促进法 第57条 | ❌ |
| Hybrid (M4) | 反电信网络诈骗法 第46条 | ❌ |
| **Two-Stage (M5)** | **循环经济促进法 第57条 (0.4982)** | ❌ |

**根因分析**:
- jieba 把"第五百七十七"切成 "五百" + "七十七"
- BM25 看到的是 "五百"、"七十七"等碎片
- Vector 把"五百七十七"当成数字概念,推荐其他法律的 5X 条
- Reranker 在 Hybrid 召回的 50 条里找,但召回的 50 条本身就没包含正确答案

**最关键的工程认知**:

> **Reranker 的分数 0.4982 远高于其他正常 query 的 0.18,
> 但实际上排第一的是错误结果.**
> Cross-encoder 会"自信地错"(over-confident on wrong answers),
> 单纯依赖 Reranker 分数做拒答判断会被骗.

→ 这是 M6 必须引入 `get_law_article` 工具的根本原因.

### 发现 5:DashScope gte-rerank-v2 分数压缩现象

实测 6 类 query 的 Top1 rerank score:

| Query | Top1 rerank |
|---|---|
| 概念性 | 0.1816 |
| 语义模糊 | 0.1831 |
| 专业术语 | 0.1929 |
| 精确条款号(错误命中) | 0.4982 |
| 多领域 | 0.1929 |
| 无关查询 | 0.1455 |

→ 正确语义查询分数普遍在 0.14~0.20 区间.
→ 但当 Reranker"误以为"找到强匹配时,分数能飙到 0.5.
→ 单一阈值法不可行,必须 多信号组合.

### 发现 6:Vector 余弦相似度 不能 跨 query 比较

| Query | Vector Top1 余弦 |
|---|---|
| 合同违约 | 0.7602 |
| 老板拖工资 | 0.7334 |
| 民法典577(错的) | 0.7225 |
| 今天天气(无关!) | 0.4588 |

→ "今天天气"的 Vector Top1 都有 0.4588.
→ 余弦相似度只反映"语义接近度",不反映"是否真正相关".
→ 这就是为什么 Vector only(M3 baseline)不能直接做拒答判断.

### 发现 7:Abstention 的工程价值不只是"防瞎编"

| Query | LLM 调用耗时 |
|---|---|
| 相关 query | 17~20 秒(完整生成) |
| 无关 query + 拒答触发 | 0 秒(直接跳过 LLM) |

→ 拒答机制不仅保证"不误导用户",还省下了 18 秒的 LLM 算力.
→ 真实生产中这意味着 大幅成本节省.

### 发现 8:延迟开销的取舍

| 阶段 | 平均延迟 |
|---|---|
| Vector only (M3) | ~600 ms |
| Hybrid (M4) | ~520 ms |
| Two-Stage (M5,纯检索) | ~1130 ms |
| Two-Stage + LLM (/chat/rag) | ~20 s |

→ Reranker 增加约 600ms 延迟(是 Hybrid 的 2 倍).
→ LLM 生成才是真正的延迟大头.
→ 法律咨询场景用户接受 1~2 秒检索延迟.

## 4. 三信号拒答策略设计

### 4.1 信号定义

| 信号 | 触发条件 | 置信度 |
|---|---|---|
| 信号 1:绝对阈值 | Top1 rerank < 0.05 | 高 |
| 信号 2:召回辅助 | Top1 rerank < 0.15 + RRF < 0.020 | 中 |
| 信号 3:领域单一 | Top5 仅命中 1~2 部边缘领域法律 | 中 |

### 4.2 实测决策路径

| Query 类型 | Top1 rerank | 决策 | 触发信号 |
|---|---|---|---|
| 概念性 | 0.1816 | 通过 | 全部信号通过 |
| 语义模糊 | 0.1831 | 通过 | 全部信号通过 |
| 专业术语 | 0.1929 | 通过 | 全部信号通过 |
| 精确条款号(错的) | 0.4982 | **误通过** ⚠️ | 分数太高,无法识别"自信地错" |
| 多领域 | 0.1929 | 通过 | 全部信号通过 |
| 无关查询 | 0.1455 | 拒答 | **信号 3:Top5 全是气象法** |

### 4.3 关键设计决策

**信号 3 必须在信号 2 之前判断**:
- 信号 2(rerank 弱 + 召回弱)是通用兜底
- 信号 3(领域单一)是 specific 信号
- specific 优先于 general,reason 输出更精准

**关键局限**:三信号无法识别"自信地错"
- 精确条款号 query 的 Top1 = 0.4982,看起来很自信
- 但实际是错误结果
- → 唯一根治:M6 引入精确查询工具

## 5. 已知局限 + 后续优化方向

### 5.1 当前局限

1. **精确条款号查询召回仍弱**
   - jieba 把中文条款号切碎导致 BM25 失效
   - Vector embedding 不区分"民法典 577"和"其他法 577"
   - **Reranker 自信地把错的排第一**(分数 0.4982)
   - 这是 Cross-encoder 在"细粒度精确匹配"上的固有局限

2. **拒答策略无法识别 over-confidence**
   - Reranker 错得自信时,拒答信号失灵
   - 需要"跨阶段一致性检查"或"专门工具"补救

3. **Reranker API 网络往返开销**
   - DashScope 调用平均 600ms
   - 可选优化:M5.x 补本地 BGE-Reranker-v2-m3

4. **拒答阈值是"经验值"**
   - 当前阈值基于 6 类 query 调出
   - 真正生产前需要标注 50+ query 跑 ROC 曲线

### 5.2 优化路线图

- **M5.x 可选**:本地部署 BGE-Reranker-v2-m3
- **M6 必做** ⭐:引入 `get_law_article` 工具,通过 Tool Use 让 LLM
  自主决策"语义检索 vs 精确查询",从根本解决 M4-M5 都无法解决的
  精确条款号问题
- **M7**:LangGraph 编排,Self-correction 循环
- **M8**:三层记忆,加多轮上下文支持
- **M11**:Neo4j 法律知识图谱,把"民法典 577"做成图节点
